The trend-detector pipeline has been running every morning for weeks. Quietly. No users. No product. Just a `briefings/` folder filling up with daily BUILD / WATCH / SKIP verdicts that I alone was reading.

At some point I realized the pipeline wasn't the interesting part. The verdict was.

The pipeline pulls 400+ rising search trends, filters ~90% through `is_buildable()`, validates pain on Reddit, checks competition via Brave, and runs Claude to produce a confidence score with sourced reasoning. Any solo dev can look at Google Trends. The thing I was actually doing — the thing I kept opening the briefing for — was the moment it said BUILD with a confidence score and a reason. That moment is the product.

## What shipped

**The validation came first.** Before writing a line of landing page code, I worked through five challenges to the concept:

- **Sources.** trendspy is the PyPI package that calls Google's internal `trending_now()` API — 400+ rising topics per call, not the unrelated French dropshipping tool. The source isn't the moat. The judgment layer is.
- **Cadence.** Daily is right for the pipeline. Four consecutive days of output show meaningfully different clusters. A 3x/week schedule would miss `trajectory` — the signal that separates a one-day spike from a sustained trend. Packaging cadence is a separate question.
- **Pricing.** $19/month at launch. Not because the value doesn't justify $39 — it does. Because at zero social proof, friction reduction matters more than margin. The plan: grandfather early subscribers, raise to $39 at 50 paying users.
- **Brand.** Amber `#F97415` as a semantic signal color, not an accent. Reserved for BUILD labels, the primary CTA, and key data points — never as decoration. Process language over "we" language: "Runs every morning at 6am" earns more trust than "We run this daily." Real briefing screenshot above the fold, not copy about the product.
- **Name.** Warranted. As in: evidence-warranted confidence. The name encodes the value prop without explaining it.

**Then the landing page.** A single-page Next.js app with the permission/confidence frame as the core positioning: *You already know what's trending. This tells you whether it's worth building — and why.* Free email signup (weekly Monday digest via Resend). Paid CTA ($19/mo Stripe Payment Link). Real verdict block visible before any copy.

The pipeline is not connected. Deliberately. The landing page exists to validate demand. If people subscribe, the briefings start flowing. If they don't, the architecture decision was right.

## Why

The product wasn't obvious from inside the pipeline. Looking at it from the outside — as a solo dev who keeps wasting weeks on the wrong problem — the valuable part was always the BUILD call, not the raw trend data. That reframe changed the positioning, the copy, the name, and the price. It didn't change a single line of pipeline code.

The hardest part of this kind of validation is that you're not testing a feature. You're testing whether you built the right thing at all.

## What's next

The landing page is live. The question now is whether anyone signs up before I connect anything else.

Runway: if paid signups hit 5, connect the pipeline and start delivering. If they don't, the landing page answered the question before the real work started.
